# TDD Runner

TDD ワークフローに従い、Jest ユニットテストの設計・作成・実行を行う。テスト失敗時のデバッグは本スキルのスコープ外とし、失敗結果を報告して終了する。

## 前提条件の確認

スキル実行前に以下を確認する:

1. `jest.config.ts` が存在するか → 存在しなければ Jest セットアップから開始
2. `package.json` に `jest`, `@types/jest` が含まれるか → なければインストールを提案

Jest セットアップの詳細は [references/jest-patterns.md](references/jest-patterns.md) の「Jest 設定」セクションを参照。

## TDD ワークフロー

### ステップ 1: テスト対象の特定

ユーザーのリクエストに応じて分岐する:

- **「この関数のテストを書いて」** → 既存コードを Read で読み、仕様を把握
- **「この機能を TDD で実装したい」** → 仕様をユーザーに確認し、テストから設計
- **「テスト設計を相談したい」** → テストケース一覧を提案し、ユーザーと合意

### ステップ 2: テストケースの設計

テスト対象を以下の観点で分析し、テストケース一覧をユーザーに提示する:

1. **正常系** — 期待入力に対する正しい出力
2. **異常系** — 不正入力、エラーケース
3. **境界値** — 空値、0、上限・下限、型境界

テストケース一覧をマークダウンで提示し、ユーザーの承認を得てから実装に進む。

```
## テストケース一覧: <関数名>

- [ ] 正常系: <説明>
- [ ] 正常系: <説明>
- [ ] 異常系: <説明>
- [ ] 境界値: <説明>
```

### ステップ 3: テストコードの作成

承認されたテストケースに基づきテストファイルを作成する。

- テストファイルは対象ファイルと同一ディレクトリに `<module>.test.ts` として配置
- テストコードのパターンは [references/jest-patterns.md](references/jest-patterns.md) を参照
- `describe` でグループ化（正常系・異常系・境界値）
- `any` は使わず型安全に記述
- インデントはタブ、セミコロンあり（プロジェクト規約に従う）

### ステップ 4: テストの実行

```bash
npx jest <テストファイルパス> --verbose
```

特定のテストのみ:

```bash
npx jest <テストファイルパス> -t "<テスト名>"
```

### ステップ 5: 結果の報告

- **全テスト通過** → 通過したテスト一覧を報告し、実装フェーズへの移行を提案
- **テスト失敗** → 失敗したテストの名前・エラーメッセージ・期待値と実際の値を報告して終了。デバッグは行わない

失敗報告フォーマット:

```
## テスト結果: FAIL

### 失敗テスト
- `<テスト名>`: <エラー概要>
  - 期待値: <expected>
  - 実際値: <received>

デバッグは本スキルのスコープ外です。実装コードの修正を行ってください。
```

## スコープの境界

| 本スキルで行うこと      | 本スキルで行わないこと         |
| ----------------------- | ------------------------------ |
| テストケースの設計      | テスト失敗時のデバッグ         |
| テストコードの作成      | 実装コードの修正               |
| テストの実行            | 統合テスト・E2E テスト         |
| 結果の報告              | テストフレームワークの選定変更 |
| Jest セットアップの確認 | React コンポーネントテスト     |
